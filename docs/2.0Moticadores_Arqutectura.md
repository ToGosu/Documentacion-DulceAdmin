## 2. Motivadores de la Arquitectura

El diseño arquitectónico de **DulceAdmin** se orienta a satisfacer tanto las **necesidades técnicas** como las **exigencias del negocio**, garantizando la mantenibilidad, escalabilidad y eficiencia del sistema.  
La arquitectura adoptada busca lograr una **separación clara de responsabilidades**, un código **modular y extensible**, y una **experiencia de usuario fluida y confiable**, incluso en entornos con conectividad limitada.

### 2.1 Restricciones Técnicas

#### 2.1.1 Restricciones impuestas
El negocio **no presenta restricciones técnicas impuestas**, por lo que el equipo de desarrollo cuenta con libertad para adoptar herramientas, tecnologías y metodologías adecuadas al contexto del proyecto.

#### 2.1.2 Restricciones adoptadas

| Tipo | Restricción | Justificación |
|------|--------------|---------------|
| **Diseño** | Se debe propender por la aplicación de prácticas basadas en *Clean Architecture* | Asegura la separación clara de responsabilidades entre capas para facilitar la mantenibilidad y escalabilidad del sistema. |
| **Diseño** | Se debe utilizar el principio de inversión de dependencias (*Dependency Inversion Principle – DIP*) | Permite que los módulos de alto nivel no dependan de módulos de bajo nivel, sino de abstracciones, mejorando la extensibilidad. |
| **Diseño** | Se debe aplicar el patrón de diseño *Observer* | Facilita la comunicación entre componentes desacoplados para manejar eventos y notificaciones. |
| **Diseño** | Se deben seguir los principios *SOLID* | Garantiza clases cohesionadas, reutilizables y abiertas a extensión, mejorando la mantenibilidad del sistema. |
| **Diseño** | Propender el uso de una arquitectura por capas | Favorece la organización modular (presentación, negocio, datos), simplificando el mantenimiento y la integración. |
| **Implementación** | Aplicar prácticas de *Clean Code* | Fomenta un código claro, legible y sostenible, con funciones pequeñas y sin duplicaciones (DRY). |
| **Implementación** | Realizar *code reviews* en cada *pull request* | Mantiene la calidad del código y facilita el intercambio de conocimiento entre desarrolladores. |
| **Implementación** | Utilizar control de versiones con Git (estrategia Git Flow) | Permite organizar el desarrollo, garantizar trazabilidad y facilitar despliegues controlados. |
| **Lineamientos** | Backend en Java con framework Spring | Ofrece robustez, modularidad, seguridad y amplia integración con bases de datos y servicios externos. |
| **Lineamientos** | Base de datos PostgreSQL | Brinda confiabilidad, consistencia y soporte para grandes volúmenes de datos, con amplia compatibilidad con Spring Data. |
| **Lineamientos** | Frontend en JavaScript con framework Vue.js | Facilita el desarrollo de interfaces modulares, reactivas y mantenibles, mejorando la experiencia de usuario. |
| **Metodología** | Uso de metodologías ágiles | Promueve la adaptabilidad, entrega continua de valor, mejora de procesos y detección temprana de errores. |

---

### 2.2 Restricciones de Negocio

Las restricciones de negocio establecen las condiciones críticas que deben ser consideradas en el diseño y operación del sistema, ya que su incumplimiento puede afectar la continuidad o reputación del negocio.

| Importancia | Riesgo | Plan de acción |
|--------------|--------|----------------|
| **Alta** | Retraso en la atención de problemas críticos, afectando la continuidad del negocio. | Establecer un contrato de soporte con tiempos de respuesta definidos, capacitar a usuarios clave y utilizar un sistema de tickets para priorizar incidencias. |
| **Alta** | Incumplimiento legal, multas y sanciones. | Mantener actualizaciones normativas, diseñar el sistema con flexibilidad fiscal y planificar actualizaciones periódicas. |
| **Media** | Pérdida o inconsistencia de datos, retrasos en inventarios o facturación. | Implementar sincronización *offline/online*, pruebas de conectividad y alertas de fallos de sincronización. |
| **Alta** | Baja adopción o uso incorrecto del sistema, aumentando errores operativos. | Capacitar a usuarios, incluirlos en pruebas y diseño, y ofrecer soporte continuo y canales de comunicación. |
| **Media** | Interrupciones en el procesamiento de pagos, afectando la experiencia del cliente. | Establecer SLA con proveedores, incluir métodos alternativos de pago y monitorear servicios externos. |
