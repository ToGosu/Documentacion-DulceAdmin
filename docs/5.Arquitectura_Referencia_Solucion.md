## 6. Arquitectura de Solución / Referencia

La **arquitectura de solución** describe cómo se concreta el arquetipo conceptual en una **implementación real con tecnologías y componentes específicos**.  
Su objetivo es mostrar **cómo se materializan las decisiones arquitectónicas** adoptadas en el proyecto *DulceAdmin*, incluyendo frameworks, librerías, servicios, herramientas y sus interacciones dentro del entorno de ejecución.

### Bloques de contruccion

| **Componente** | **Justificación** |
|----------------|-------------------|
| **Notification Gateway** | Centraliza el envío de notificaciones, separando la capa de mensajería del backend principal. Esto mejora la escalabilidad y la resiliencia, permitiendo manejar colas de mensajes y asegurar la entrega incluso ante fallos parciales. |
| **PaaS** | El PaaS abstrae la infraestructura subyacente, permitiendo enfocarse en el desarrollo sin preocuparse por servidores, sistemas operativos o escalado manual. Facilita el despliegue continuo (CI/CD), la observabilidad y la recuperación ante fallos, reduciendo los costos operativos y de mantenimiento. |
| **Identify Management** | Este componente permite administrar las identidades digitales de los usuarios y sus permisos dentro del sistema. Es clave para garantizar que solo usuarios legítimos accedan a los recursos, centralizando la gestión de credenciales y roles. |
| **RestAPI** | Define la interfaz pública de la lógica de negocio, normalmente siguiendo el estilo REST para interoperabilidad. Permite que distintos clientes consuman los servicios de forma uniforme, desacoplando el frontend del backend. |
| **Backend** | Es el núcleo de la solución: procesa peticiones, aplica validaciones, gestiona transacciones, coordina el acceso a datos y orquesta servicios externos. |
| **Instrumentation / Traceability** | Proporciona visibilidad sobre el funcionamiento de los servicios: métricas de rendimiento, logs, trazas distribuidas y alertas. Es fundamental para la observabilidad, el diagnóstico de errores, la auditoría y el cumplimiento de normativas. |
| **DataBases** | Las bases de datos permiten guardar información crítica del negocio (usuarios, transacciones, inventario, etc.) de forma segura y consistente. Forman la fuente de verdad del sistema y deben residir en una red privada protegida, con respaldo y replicación. |
| **Payment Gateway** | Permite realizar transacciones económicas seguras mediante la conexión con proveedores de pago. Aísla la lógica financiera del backend y reduce riesgos al delegar la gestión de datos sensibles a servicios certificados PCI-DSS. |
| **Blob Storage** | Permite guardar imágenes, documentos, facturas u otros archivos grandes que no deben almacenarse en la base de datos. Ofrece alta disponibilidad, durabilidad y bajo costo. |
| **API Gateway** | El API Gateway canaliza todas las solicitudes hacia los distintos servicios del backend, gestionando aspectos como autenticación, enrutamiento, control de tráfico, monitoreo y límites de tasa. Facilita la observabilidad, la seguridad y la escalabilidad horizontal del sistema. |
| **WAF** | El WAF actúa como una barrera de seguridad entre Internet y la aplicación, filtrando tráfico malicioso. Es esencial en cualquier solución expuesta públicamente para garantizar la integridad y disponibilidad del sistema, reduciendo la superficie de ataque. |
| **Frontend** | El frontend permite a los usuarios acceder a las funcionalidades del sistema mediante una interfaz gráfica. Este componente se comunica con los servicios backend a través de la red. |


En esta arquitectura, los componentes se implementan siguiendo los lineamientos definidos en el arquetipo, pero con asignación concreta de tecnologías y responsabilidades.  
A continuación, se detalla la arquitectura de referencia empleada:

| Componente Concreto | Intención / Función | Justificación / Uso |
|----------------------|---------------------|----------------------|
| **Frontend (Vue.js)** | Interfaz visual e interactiva del sistema. | Vue.js permite desarrollar interfaces reactivas, modulares y mantenibles, con excelente rendimiento en entornos locales o web. |
| **Backend (Spring Boot – Java)** | Implementa la lógica de negocio, controladores y servicios. | Spring Boot facilita la creación de aplicaciones robustas y escalables, integrando seguridad, persistencia y REST APIs. |
| **Base de Datos (PostgreSQL)** | Almacenamiento estructurado de la información. | PostgreSQL garantiza consistencia, transacciones seguras y compatibilidad con frameworks Java mediante Spring Data JPA. |
| **Módulo de Sincronización Local** | Maneja la operación offline y sincronización con la base de datos principal. | Permite continuidad de operaciones sin conexión, asegurando integridad de datos al reconectarse. |
| **Módulo de Seguridad y Control de Acceso** | Autenticación y gestión de roles (mesero, administrador, contador). | Protege el acceso a funciones críticas mediante control de permisos y cierre automático de sesión. |
| **Módulo de Ventas y Facturación** | Gestión de transacciones, generación de recibos y registro de ventas. | Constituye el núcleo operativo del sistema, garantizando fluidez y precisión en el proceso de venta. |
| **Módulo de Inventario** | Administración del stock de productos, registro y actualización de existencias. | Permite control de inventario en tiempo real, reduciendo errores y pérdidas operativas. |
| **Módulo de Reportes** | Generación de reportes de ventas, inventario y desempeño. | Ofrece información consolidada para la toma de decisiones, optimizando el control administrativo. |
| **Sistema Operativo y Red LAN** | Entorno de ejecución local. | Permite funcionamiento offline y comunicación interna rápida entre dispositivos. |

Esta arquitectura está diseñada para ser **flexible, escalable y mantenible**, permitiendo futuras integraciones con servicios en la nube, aplicaciones móviles o sistemas de terceros.
